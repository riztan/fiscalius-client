Manual del Programador

Guía técnica completa para desarrolladores que trabajan con Fiscalius.

Introducción

Este manual está dirigido a desarrolladores que desean entender, modificar o extender Fiscalius. El sistema está desarrollado en TPuy (framework basado en Harbour + T-GTK) y utiliza una arquitectura cliente-servidor.

1. Arquitectura del Sistema

1.1 Diagrama de Arquitectura

┌─────────────────────────┐   hb_netio TCP/IP   ┌───────────────────────┐   SQL   ┌────────────┐
│  Fiscalius Client       │ ←─────────────────→  │ Fiscalius Server      │ ←─────→ │ PostgreSQL │
│  (TPuy/xHarbour + GTK)  │                       │ (Harbour/TPuy)        │         │ 16         │
│  UI + Validaciones prev.│                       │ API + Lógica negocio  │         └────────────┘
└─────────────────────────┘                       └───────────────────────┘

1.2 Componentes

  • Cliente: Interfaz gráfica GTK+ desarrollada en TPuy
  • Servidor: API RESTful desarrollada en Harbour/TPuy
  • Base de datos: PostgreSQL 16
  • Comunicación: Protocolo NetIO (hb_netio)

2. Estructura del Proyecto

3. Tecnologías Utilizadas

3.1 TPuy Framework

TPuy es un framework para desarrollo de aplicaciones de escritorio con interfaces GTK+. Está basado en:

  • Harbour: Compilador de xBase compatible con Clipper
  • T-GTK: Binding de Harbour para GTK+
  • hb_netio: Biblioteca de comunicación cliente-servidor

3.2 PostgreSQL

Sistema de base de datos relacional utilizado para almacenar:

  • Datos de la empresa
  • Catálogo de productos
  • Clientes y proveedores
  • Facturas y documentos fiscales
  • Movimientos de caja
  • Inventario

4. Patrones de Código

4.1 Comunicación con el Servidor (NetIO)

// Verificar conexión primero
IF !oTPuy:RunXBS("netio_check")
   MsgStop("Sin conexión al servidor")
   RETURN
ENDIF

// Obtener objeto remoto
::rFinanzas := oFormParent:oGEmpresa:SetModulo("Finanzas")
::rCobranza := ::rFinanzas:Cobranza('')

// Verificar respuesta
IF hb_isObject( ::rCobranza )
   hData := ::rCobranza:Data()
ENDIF

4.2 Cargar Datos a ListBox

oDatos := rEmpresa:Metodo( hParams )
oModel:oTreeView:ClearModel(.T.)
FOR EACH aLine IN oDatos:GetData()
   APPEND LIST_STORE oModel:oLbx ITER aIter VALUES aLine[1], aLine[2], aLine[3]
NEXT

4.3 Obtener Valor de Celda

nId    := oTView:GetAutoValue( 1, aIter )   // Columna 1
cNombre:= oTView:GetAutoValue( 2, aIter )   // Columna 2

4.4 Conversión Numérica

// ✅ CORRECTO - mantiene decimales
nValor := ToNum( cValorCampo )

// ❌ INCORRECTO - VAL() puede perder decimales
nValor := VAL( cValorCampo )

4.5 Variables Locales

// LOCAL debe declararse AL INICIO del procedimiento
PROCEDURE MiProc()
   LOCAL aLine, nLen, aVal
   FOR EACH aLine IN aData
      nLen := LEN( aLine )
      ...
   NEXT
RETURN

5. Objetos Remotos

5.1 Objeto Empresa

::rEmpresa := oFormParent:oGEmpresa
::rVentas := ::rEmpresa:SetModulo("Ventas")
::rFinanzas := ::rEmpresa:SetModulo("Finanzas")
::rInventario := ::rEmpresa:SetModulo("Inventario")

5.2 Métodos Comunes

  • SetModulo(cModulo): Obtiene un módulo remoto
  • Clientes(hParams): Lista de clientes
  • Facturas(cId): Instancia de factura
  • Documento(cTipo, cNumero): Crea documento
  • CxCxFlujo(hParams): Cuentas por cobrar
  • Cobranza(cId): Instancia de recibo

6. Reglas Importantes

6.1 Reglas Absolutas

  • NUNCA usar VAL() - usar siempre ToNum()
  • NUNCA modificar archivos .xbs o .ui sin permiso
  • NUNCA hacer commits sin autorización
  • NUNCA usar valores hardcoded
  • SIEMPRE verificar coherencia: BD = API = Cliente
  • SIEMPRE verificar objetos antes de acceder
  • SIEMPRE cerrar queries y limpiar recursos

6.2 Regla de Tiempo

Si en ~1 minuto no encuentras la información que necesitas, DETENTE y pregunta al usuario.

7. Contribuir al Proyecto

7.1 Flujo de Trabajo

  1. Forkear el repositorio
  2. Crear una rama para tu feature o fix
  3. Desarrollar y probar tus cambios
  4. Crear un Pull Request con descripción clara
  5. Esperar review y merge

7.2 Normas de Código

  • Sigue el estilo de código existente
  • Agrega comentarios solo cuando sea necesario
  • Prueba tus cambios antes de enviar
  • Documenta las funciones nuevas
